# Deterministic fuzzing

Development version 0.1.0-dev includes Ada-native fuzzing infrastructure for robustness testing. The
fuzzers are deterministic: every generated byte stream, mutation, chunk split,
payload pattern, and lifecycle sequence is derived from an explicit 32-bit seed
and iteration count. A failing run should be reproducible by rerunning the same
target with the same seed and iteration count.

Fuzzing is a hardening aid, not a replacement for release correctness tests.
Normal correctness remains covered by the AUnit suite and conformance fixtures.
Heavy fuzzing is manual; short deterministic smoke passes are run by `tools/bin/check_all`
after the ordinary build/test/example path.

## Scope

The fuzzing infrastructure targets:

- zlib, gzip, and raw Deflate one-shot inflate paths;
- streaming inflate chunk boundaries and tiny output buffers;
- compression-to-inflate roundtrips for patterned binary, repeated, text-like,
  dictionary-like, ramp, and random payloads;
- stored, fixed, dynamic, and auto compression modes;
- compression levels 0 through 9 across zlib, gzip, and raw wrappers;
- gzip metadata fields including FEXTRA, FNAME, FCOMMENT, MTIME, OS, XFL, and FHCRC;
- implicit multi-member gzip inflate policy and explicit strict single-member policy;
- preset dictionary roundtrips and dictionary mismatch handling;
- bit flips, truncation, byte duplication/removal, and trailer/header corruption;
- streaming compression flush sequences across wrappers, modes, chunk sizes, and
  tiny output buffers;
- invalid streaming lifecycle sequences such as calls before init, after close,
  dictionary changes after data start, and sync/full flush after stream end;
- flush handling and finish handling with partial input/output buffers;
- tiny-buffer decoding of raw, zlib dynamic, gzip metadata, gzip multi-member,
  and full dictionary-backed streams.

The fuzzing package and tools do not add public compression behavior. They are
release hardening infrastructure only.

## Package boundary

The drivers use `Zlib.Fuzzing`, an infrastructure child package. Normal consumer
programs should continue to use only the public root package `Zlib`. The child
package exists so tools and AUnit smoke tests share the same deterministic
sequence generator, mutation helpers, target dispatch, result digest, and
acceptance policy.

## Tools

The tool project builds these fuzz drivers:

```text
tools/fuzz/fuzz_inflate.adb
tools/fuzz/fuzz_streaming_inflate.adb
tools/fuzz/fuzz_compress_roundtrip.adb
tools/fuzz/fuzz_compress_levels.adb
tools/fuzz/fuzz_wrapper_mix.adb
tools/fuzz/fuzz_dictionary.adb
tools/fuzz/fuzz_gzip_metadata.adb
tools/fuzz/fuzz_mutation.adb
tools/fuzz/fuzz_flush.adb
tools/fuzz/fuzz_lifecycle.adb
tools/fuzz/fuzz_tiny_buffers.adb
```

Build them with:

```sh
alr exec -- gprbuild -P tools/tools.gpr
```

Each driver accepts optional positional arguments:

```text
<iterations> <seed>
```

Example manual runs:

```sh
./tools/bin/fuzz_inflate 500 63
./tools/bin/fuzz_streaming_inflate 500 64
./tools/bin/fuzz_compress_roundtrip 500 630
./tools/bin/fuzz_compress_levels 500 634
./tools/bin/fuzz_mutation 500 631
./tools/bin/fuzz_lifecycle 100 632
./tools/bin/fuzz_tiny_buffers 500 633
./tools/bin/fuzz_wrapper_mix 500 635
./tools/bin/fuzz_dictionary 250 636
./tools/bin/fuzz_gzip_metadata 250 637
./tools/bin/fuzz_flush 500 638
```

## Target matrix

| Driver | Main coverage | Expected deterministic failures allowed |
| --- | --- | --- |
| `fuzz_inflate` | arbitrary one-shot zlib/gzip/raw inflate input | yes |
| `fuzz_streaming_inflate` | arbitrary streaming inflate input and chunking | yes |
| `fuzz_compress_roundtrip` | generated payloads, modes, wrappers, roundtrip verification | no |
| `fuzz_compress_levels` | levels 0..9 across supported wrappers | no |
| `fuzz_wrapper_mix` | wrapper separation, mismatch rejection, gzip multi-member input | no |
| `fuzz_dictionary` | dictionary roundtrips and wrong-dictionary failures | no |
| `fuzz_gzip_metadata` | metadata header fields and gzip roundtrips | no |
| `fuzz_mutation` | corrupted zlib/gzip/raw streams and deterministic rejection | yes |
| `fuzz_flush` | valid streaming compression flush sequences and roundtrips | no |
| `fuzz_lifecycle` | invalid streaming inflate/compress lifecycle sequences | no |
| `fuzz_tiny_buffers` | one-byte and tiny-buffer streaming decode paths | no |

The summary line includes the target, seed, iteration count, deterministic
success/failure counts, unexpected crash count, digest, last input length, and a
hex snippet. A nonzero crash count is always a fuzz failure. Deterministic
non-`Ok` results are allowed only for the negative-input targets that
intentionally feed arbitrary or corrupted streams: `fuzz_inflate`,
`fuzz_streaming_inflate`, and `fuzz_mutation`. Roundtrip, wrapper, dictionary,
metadata, flush, lifecycle, level, and tiny-buffer targets fail when any
deterministic failure is reported.

## Reproducibility

The internal generator uses a fixed linear congruential step:

```text
Seed := Seed * 1664525 + 1013904223
```

Do not replace it with an external RNG for normal fuzzing. If a failure appears,
record:

```text
target
seed
iteration count
wrapper/mode represented by the target
summary digest from the tool output
last input length and hex snippet from the tool output
```

When the failure is a real library bug, add a minimized regression fixture to
the normal AUnit suite. Long-running fuzz logs are not a substitute for a small
stable regression test.

## Corpus policy

`tests/fuzz_corpus/` is reserved for minimized deterministic fixtures found by
fuzzing. It should contain small, stable inputs plus a short note explaining the
bug each fixture prevents. Do not add generated random corpora, large binary
logs, or machine-local reproducer output.

Corpus fixtures are stored as ASCII hex files and replayed by the AUnit fuzz
smoke suite. The initial replay set covers valid empty zlib, gzip, and raw
Deflate streams plus a truncated zlib header. Add new fixture files only with a
matching executable assertion in the normal test suite.

## Optional system zlib cross-checking

System zlib cross-checking is optional. The crate must not depend on an external
zlib installation for normal builds, tests, or release validation. Any future
system-zlib comparison tool must remain outside the required release path unless
that dependency decision is explicitly revisited.
